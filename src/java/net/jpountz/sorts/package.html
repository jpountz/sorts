Sorting algorithms.

Comparison

{@link net.jpountz.sorts.Sorter}RuntimeMemoryStableNotes
{@link net.jpountz.sorts.BinarySorter} O(n2) O(1) Yes Only for small arrays.
{@link net.jpountz.sorts.HeapSorter} O(n ln(n)) O(1) No
{@link net.jpountz.sorts.InPlaceMergeSorter} O(n ln(n)) O(1) Yes O(n) on sorted arrays.
{@link net.jpountz.sorts.InsertionSorter} O(n2) O(1) Yes Only for small or almost sorted arrays.
{@link net.jpountz.sorts.IntroSorter} O(n ln(n)) O(1) No Outperforms other implementations on randomly-sorted data.
{@link net.jpountz.sorts.MergeSorter} O(n ln(n)) n Yes O(n) on sorted arrays, several times faster than {@link net.jpountz.sorts.InPlaceMergeSorter} on randomly-sorted data.
{@link net.jpountz.sorts.TernaryHeapSorter} O(n ln(n)) O(1) No Slightly faster than {@link net.jpountz.sorts.HeapSorter} on large arrays.
{@link net.jpountz.sorts.TimSorter} O(n ln(n)) Configurable, up to n/2 Yes O(n) on partially-sorted data, only slightly slower than {@link net.jpountz.sorts.MergeSorter} on randomly-sorted data.

Which implementation to use?

  • If your data is likely partially sorted, then use {@link net.jpountz.sorts.TimSorter}.
  • Otherwise if you don't need the sort to be stable, then use {@link net.jpountz.sorts.IntroSorter}.
  • Otherwise if you can afford high memory usage, then use {@link net.jpountz.sorts.MergeSorter}.
  • Otherwise use either {@link net.jpountz.sorts.TimSorter} (faster) or {@link net.jpountz.sorts.InPlaceMergeSorter} (easier to implement).
